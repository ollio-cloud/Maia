4. Design / Traffic Flow

Primary/DR Architecture: Azure East is the primary region for production traffic. Azure South East is the DR region (pre-staged, not active under normal operation).

Route Table Isolation: KD-Prod-RouteTable is associated only with East workload subnets; KD-Mel-RouteTable only with SEast workload subnets. Each region routes to sites via its local FortiGate. This prevents asymmetric routing during normal operation.

Why routes in both regions: Site subnets are pre-staged in both route tables so that, during a DR event, workloads failed over to SEast retain connectivity to all site subnets via the SEast FortiGate.

DR Failover: Manual failover (not automatic). DR is invoked by Azure workload failover (ASR/manual), not by this VPN configuration. DR failover testing is not in scope for this CR; DR readiness is achieved by pre-staging configuration only.

9. Test Plan (Per Site)

Normal State Acceptance:

  • Site → Azure East (Primary): Ping from site to 10.200.10.10 - expect reply
  • Azure East → Site: Ping from Azure East VM to site subnet - expect reply
  • RDP to Azure East VMs to confirm bidirectional traffic (no asymmetric routing)
  • Verify East FortiGate tunnel stats incrementing (production traffic)

DR Pre-staging Validation:

  • Site → Azure SEast (DR): Ping from site to 10.201.10.61 - expect reply (tunnel up)
  • Verify SEast FortiGate tunnel established (idle/standby - no production traffic expected)
  • Note: DR failover testing (forced East tunnel down) is not in scope for this CR

Isolation Check:

  • Confirm other sites unaffected (Meraki tag changes are staged per batch)

10. Rollback Procedure (Per Site)

If tests fail: Execute in order (estimated < 5 minutes per site)
  1. Remove site routes from KD-Prod-RouteTable (Azure East)
  2. Remove site routes from KD-Mel-RouteTable (Azure SEast)
  3. Disable VPN tunnels on both FortiGates
  4. Verify site connectivity via existing SonicWall
Why both regions: Rollback removes both primary and DR paths to force reversion to SonicWall and prevent partial failover/asymmetric routing.
SonicWall Fallback: SonicWall configuration remains unchanged until completion of all batches + 14 days soak period.
