Post Incident Review - CR-4378310

Incident Summary

Change RequestCR-4378310 - KD Bus Sites Production VPN Migration
Related CRCR-4342297 (Pilot Test)
ImpactSome on-site users unable to access Azure resources
DurationResolved during pilot phase

What Happened

During the pilot phase, the existing Meraki network tags KDBus-Azure-VPN and KDBus-Azure-DR-VPN were applied to the new FortiGate IPsec VPN peers. These tags were already in use by the existing SonicWall VPN peers, which caused both the SonicWall and FortiGate tunnels to become active simultaneously for the tagged networks. This created a dual-tunnel routing conflict, resulting in some on-site users being unable to access Azure resources.

Root Cause

Applying shared network tags to both old (SonicWall) and new (FortiGate) IPsec peers caused Meraki to route VPN traffic across both tunnels concurrently. The conflicting paths led to asymmetric routing and connectivity failures for Azure-hosted resources.

Resolution

A dedicated network tag (fortigate-test) was created and applied exclusively to the new FortiGate IPsec peers. This isolated the new tunnels to only the pilot network, while all other sites continued routing via the existing SonicWall VPN unaffected. The tag restriction approach was adopted as the standard method for the phased production rollout.

Lessons Learned

  • New IPsec VPN peers must use dedicated/unique network tags to avoid conflicts with existing VPN tunnels during migration.
  • Meraki network tags are the primary mechanism for controlling which networks use which IPsec peers — shared tags will activate both old and new tunnels simultaneously.
